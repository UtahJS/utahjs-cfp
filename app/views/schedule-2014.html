Stateful AJAX Authentication Without Server-side Sessions

Standard web applications handle authentication with server side sessions and cookies. That works fine until you hit scale, at which point storing those sessions becomes difficult, as does routing through load balancers to the right server cluster that handled your earlier requests. We'll discover a secure way to store that information on the client without exposing API keys, while retaining control of session expiration, that will work with any front-end or backend configurations. While the reasons for doing this are complex, the actual implementation is something that even beginner JS developers will be able to comprehend and implement.

Targeted for: Beginner Developers, Experienced Developers

Functional Javascript: The Good Parts

Functional programming encourages us to organize our code well and helps us create code that can be easier to reason about. The less we reinvent the wheel and use a well-tested tool, the easier it is to maintain our application code.

"Pure" functions return a result based only on what you pass in. You never have to worry it is going to do something unexpected. Higher order functions (functions that take other functions as parameters) let us reuse code in places we wouldn't expect, like loops. Composition allows you to create larger tools from smaller ones.

Javascript is a great functional language, but some ideas are harder to implement than others, and some aren't worth using in JS at all! After having wandered the Functional JS wastes for a few years, I will tell you stories of the dangers that lurk, as well as the rewards that await you if you apply Functional JS wisely.

i18n with AngularJS

i18n is a part of development that is often overlooked. While challenging to implement, when done properly i18n can drive more traffic to your site and provide more value to your users.

My talk will cover the following:

What is i18n and why is it important?

I am constantly surprised by how many developers don’t even know what i18n stands for. Beyond describing what i18n is, I will discuss the value that it can provide to your site and users.

How to detect locale

Before you can provide locale specific content, you need to know what locale your user prefers. I will show how this detection can be done.

Serving translated pages

If a user visiting your site only speaks french, but your site only serves pages in english they are not going to stick around beyond the first page. This has a negative impact on your business, especially if you offer a product or service they would otherwise be interested in, but they couldn’t read your site to find it. I will discuss how to serve translated pages with AngularJS.

Formatting data

Aside from translating text content, some data needs to be formatted differently based on locale. Specifically dates, numbers and currency. AngularJS provides filters to manage this for you. I will show how these filters are used.

Generators and Koa: A Modern Approach to Flow Control and Middleware in Node

Generators are a new JavaScript feature from the ES6 Harmony specification that are already available in Node 0.11.x. With generators, we can achieve more elegant asynchronous flow control than is possible with callbacks alone.

Luckily, some of the authors of the popular Express framework saw the potential of generators from day one and created Koa, a "next generation web framework" that leverages generators to vastly improves the typical approach to middleware.

The first half of this presentation will provide a comprehensive explanation of how generators work and how they can be used to write cleaner, clearer asynchronous code. The second half will be an introduction to Koa.

Do you REALLY know JavaScript?

As I talked about in my introduction, I'm very heavily involved in teaching JavaScript. Sadly, JavaScript usually isn't traditionally taught at the University level. Although most engineers have experience with more traditional programming languages like C++ and Java, JavaScript as we know is quite different. With the rise of frameworks like Angular and Ember ( +many more), fundamental JavaScript programming is being abstracted away. This is bad, and we should feel bad. I'm amazed at the amount of engineers I see who don't REALLY know what the 'new' keyword does in JavaScript, or those who still aren't confident about what 'this' in JavaScript actually does. In my talk I'm hoping to kick it old school Crockford style. I'll quickly talk the beginnings of JavaScript, then go straight into fundamentals. new, this, call, apply, bind, instantiation patterns, etc etc etc. If time permits, I'd love to cover the future of JavaScript as well comparing what we have now, to what's going to change with ES6. I really love Angular, but the reason I love it is because I'm able to understand it from a vanilla JavaScript viewpoint, I'd like more people to experience that.

Open Source is Best Practice

When writing software we decide where to put code a hundred times a day. Usually the easiest place to drop it is the hardest to maintain, but it's not always obvious what to do. @substack once said something like: "Whenever I approach a problem I think: what tools, if they existed would make this problem trivial?" Using open source means you don't have to reinvent the wheel, and it reduces the complexity of your application code.

But what about when the library you need doesn't exist? This talk centers on the idea that we should think of our applications as a small amount of application code pulling together many well-designed open source libraries. We all benefit from more well-designed open source, but even if your team never publishes anything, if you extract all the generic parts of your application into focused libraries, your code base will end up well-factored by default.

We will also discuss good design principles for open source libraries, the tools available to us to make it easier, and why it is a good business decision too.

Typescript is testing made easy

Typescript is a minimal superset of Javascript that adds a modern type system and ES6 features.

Testing your code has never been more popular. It prevents you or other people on your team from breaking things that already work. Most importantly, it gives you the confidence to edit and refactor legacy code. However, testing is difficult. We must maintain tests, they are difficult to write, and they are slow to write and slow to run. Strict typing gives your Javascript similar benefits but is much easier to implement. It's like putting your tests inline with your code while you write it. You can see your mistakes inline every time you save.

Typescript adds a modern type system to Javascript that is 20 years better than Java. It doesn't get in your way, and allows you to catch many complex errors. It's extremely easy to add to your project since everything is optional and all JS is valid. It compiles to clean, semantic Javascript. It also adds a few ES6 features like imports, fat-arrow functions, and classes.

In this presentation I will show what it is like to work with Typescript on a real project and how easy it is to implement.

Ending the async debate: ES6 Generators

I recently had the chance to play around with Koa.js, a node server-side framework that introduced me to the subject of ES6 generators. They solve Javascript's async problem very nicely and they're just around the corner. They are currently supported in Node 0.11, Firefox 26 and Chrome 29.

Generators allow you to write special functions that can yield control to the outside. They are useful for iterators, async, and many other things too. When used together with promises they allow you to write sequential async code completely naturally, as if it were synchronous.

We will compare Javascript async patterns, including the callback pyramid of doom, promises, and finally using promises with generators.
